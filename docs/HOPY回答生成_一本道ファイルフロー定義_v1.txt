----HOPY回答生成 一本道ファイルフロー定義 v1 ここから----

【定義名】
HOPY回答生成 一本道ファイルフロー定義 v1

【目的】
HOPY回答生成で不具合調査・機能追加・推論エンジン接続を行うときに、
ユーザー入力からHOPY回答、assistant message保存、hopy_confirmed_payload、Future Chain、Learning、MEMORIES、最終payloadまで、
どのファイルをどの順番で通るかを一目で追えるようにする。

この定義の目的は、
原因調査時に横道へそれず、
「今どこで止まっているか」
「どのファイルの責務か」
「どこを触ってはいけないか」
を明確にすることである。

【最重要原則】
HOPY回答確定後の唯一の正はこれ。

hopy_confirmed_payload

HOPY回答生成後の下流で再判定してはいけないもの。

state_changed
state_level
current_phase
prev_state_level
prev_phase
Compass表示可否
HOPY回答○表示可否
Future Chain delivery_mode
owner_handoff / recipient_support の発生理由
HOPY回答本文の意味
Compass本文の意味

回答前は HOPY DB Brain / Learning / MEMORIES / prompt materials を読む。
回答後は hopy_confirmed_payload が唯一の正になる。

回答後の下流機能は、
hopy_confirmed_payload と確定済み confirmedTurn を受け取り、
保存・表示・中継・履歴化するだけであり、
意味を作り直してはいけない。

【大前提】
状態値は必ず 1..5。

1 = 混線
2 = 模索
3 = 整理
4 = 収束
5 = 決定

0..4 へ戻してはいけない。

【Phase 1：回答前 context / prompt 準備】
役割：
ユーザー入力、履歴、MEMORIES、Learning prompt材料を読み、HOPY回答生成に必要な prompt input を作る。

中心ファイル：

buildPromptInput()
authenticatedLearning.ts
loadLearningPromptContext()
listLearningRecordsForPrompt.ts
buildLearningPromptContext.ts
buildLearningInjectionBlock.ts

責務：
- conversation history を読む
- prompt input を作る
- Learning DB から回答前材料を取得する
- LearningPromptContext を作る
- Learning injection block を作る

やらないこと：
- hopy_confirmed_payload を作らない
- state_changed を確定しない
- Compassを生成しない
- Future Chainを保存しない
- MEMORIESを保存しない
- UI表示をしない

やらないこと：
- DB保存しない

- Future Chain保存しない
- Learning保存しない

HOPY回答と状態値を confirmedTurn として扱える形へ整える。

【不具合調査時の確認順】
HOPY回答生成で不具合が出た場合は、この順に見る。

1. runTurnResult があるか
2. confirmedTurn が正しく同期されているか
3. state_changed / Compass invariant で止まっていないか
4. assistantMessageId が messages insert returning id として取れているか
5. buildFutureChainContextFromFinalizedTurn() が context を作っているか
6. saveFutureChainFromConfirmedPayload() が save / skip / error のどれを返したか
7. payload.hopy_confirmed_payload が載っているか
8. payload.hopy_confirmed_payload.state が 1..5 か
9. payload.hopy_confirmed_payload.future_chain_context が載っているか
10. payload.future_chain_persist が載っているか
11. payload.future_chain_display が載っているか
12. recipient_support の場合、selectFutureChainRecipientSupport() が selected を返したか
13. recipient_support の場合、futureChainDeliveryEvent.ts が deliveryEventId を返したか
14. UI側が payload.future_chain_display を描画しているか

【触ってはいけない方針】
不具合が出ても、以下はしない。

- state_changed を下流で作り直す
- state_level を 0..4 に戻す
- Compassがないから補う
- Future Chainが出ないからUI側で無理やり表示する
- hopy_confirmed_payload 以外から表示正を作る
- HOPY回答本文をFuture Chain側で再要約する
- Compass本文をFuture Chain側で再要約する
- memory candidates をpayload復元地点で意味判定する
- authenticatedPostTurn.ts に分離済み処理を戻す
- authenticatedPayload.ts に top-level reply/state/compass を戻す
- futureChainDisplayPayload.ts に recipient_support検索を入れる
- futureChainDeliveryEvent.ts に表示候補選択を入れる
